PHASE 3 — WAIT
--------------------------------------------------------------------------------

STEP 6 — Halt and wait for an explicit decision
    After summarizing, STOP. Do not guess, do not "just try" anything.
    Expect Arvin to reply with ONE of:

      "rollback to <REV>"          → Phase 4.A
      "forward-fix: <short plan>"  → Phase 4.B
      "dig deeper: <what to check>" → loop back to Phase 1 with new commands
      "stand down"                 → Phase 5 (false alarm or resolved)

    If Arvin's reply is ambiguous, ASK before acting.

--------------------------------------------------------------------------------
PHASE 4 — ACT
--------------------------------------------------------------------------------

STEP 7A — Rollback (traffic flip, no code change)
    Confirm the target revision exists and is Ready:
        gcloud run revisions describe <LAST_GOOD_REV> \
          --region=us-central1 --project=fis-lead-gen \
          --format='value(status.conditions)'
    If status shows True/Ready, flip traffic:
        gcloud run services update-traffic {{AFFECTED_SERVICE}} \
          --region=us-central1 --project=fis-lead-gen \
          --to-revisions=<LAST_GOOD_REV>=100
    Takes ~30 s. Do not mix with other changes in this command.

STEP 7B — Forward-fix (code change on develop → promote to main)
    This is a normal feature flow with extra urgency. Use the backend-solo
    template's Steps 0–9 to ship the fix to develop, then open a DEVELOP→MAIN
    release PR to promote it to production. For a HOTFIX that should skip
    develop entirely (rare), branch from `main` as `hotfix/<slug>`, PR to
    `main`, merge with --admin, then `gh pr create --base develop --head main`
    to sync develop back. Discuss with Arvin before picking the hotfix path.

    Hard checkpoints in forward-fix:
      - Re-read plans/ and reports/ for any prior context.
      - Write a failing test that reproduces the incident BEFORE writing the fix.
      - Manual smoke test (curl) against staging before promoting to main.
      - Keep commits and PR body in Arvin's voice; no AI attribution.

STEP 7C — Dig deeper
    Run the new commands Arvin specified. Loop back to Step 4 (append to
    report), Step 5 (summarize), Step 6 (wait).

--------------------------------------------------------------------------------
PHASE 5 — VERIFY + CLOSE
--------------------------------------------------------------------------------

STEP 8 — Confirm recovery
    After any remediation, re-probe the symptoms that defined the incident:
      - curl the affected endpoint(s); expect 200 / 401 / 403 as appropriate,
        NOT 500 / 502 / 503.
      - Re-pull error logs for the now-serving revision; they should be quiet.
      - For a rollback: verify `status.traffic` shows 100% on the rollback
        revision (`gcloud run services describe`).

STEP 9 — Update the incident report with resolution
    Append to {{REPORT_PATH}}:

        ## Resolution — $(date -u +"%Y-%m-%dT%H:%M:%SZ")

        Action taken: <rollback to rev X / merged hotfix PR #N / stood down>
        Wall-clock to recovery: <minutes>
        Customer impact: <best estimate>
        Verification: <commands run + results>

        ## Follow-ups

        - <each: owner + priority>
        - Post-mortem? yes/no (yes for P0/P1)

STEP 10 — Chat summary (≤ 5 lines)
    - What was broken, what we did, how long, whether prod is green.
    - Link to the updated incident report.
    - If a post-mortem is needed, say so — but don't start writing it
      without Arvin's go-ahead.

--------------------------------------------------------------------------------
ANTI-PATTERNS (things that turn a small incident into a big one)
--------------------------------------------------------------------------------
- "Let me just try restarting it."  → No. Investigate first.
- "I'll rollback AND ship a fix."    → One remediation at a time. Rollback,
                                       verify green, THEN forward-fix on a
                                       separate PR.
- "I'll deploy directly to production to save time." → No. Staging first,
                                       even in incidents, unless Arvin
                                       explicitly says skip-staging.
- "The logs look empty so it must be fine now." → Empty logs on a broken
                                       container means the container isn't
                                       running far enough to log. Investigate
                                       harder, don't stand down.
